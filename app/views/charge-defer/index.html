Charge - deferral for consent to restriction

When an application contains a restriction which requires the consent of a specified party. If the consent is missing a requisition needs to be sent and the application deferred (put into a specific pot whilst we wait for a response to the requistion).

A caseworker:

  • will know if consent is required
  • will write a requisition letter (C90), if this is the only reason to requisition
  • use legacy system PC Corres to write the letter

The system will:

  • add a PAS note, allowing the application to be stored until the consent is received
  • put the application in a new 'pot' (category of work) to be picked up for evidence to be checked
  • run all the business rules again, to see if anything else has changed

Round 2 prototypes for user research

For the second round we are testing iterations of some elements of the design. These include new versions of the restrictions question which should clarify which restrictions it is referring to and updating the deferral confirmation screen to no longer mention workflow as that had caused confusion. We also want to test new scenarios to establish when certain screens are misaligned with caseworkers expectations despite accurately representing the systems actions.

Round 1 prototypes for user research

From these scenarios we hoped to learn about the relationship between both deferral questions (charge evidence and restrictions), how users understood deferrals in charges, and how they navigate and review restrictions on the register.

  • A - Both UI tasks
    Scenario A: The register contains a charge with a restriction and another type of restriction. The other type of restriction does not require consent. The charge evidence is missing. The application should defer with the charge evidence, however our assumption is that it won't due to both tasks being present and caseworkers answering with workarounds.
  • B - Two restrictions
    Scenario B: One restriction requires consent, one does not. Consent has not been provided for the restriction that requires it and the case should therefore defer.
  • C - Multiple restrictions
    Scenario C: Multiple restrictions, some needing consent and others not. Consent has been provided for all that require it and therefore the case should markoff.
  • D - Referral
    Scenario D: One restriction that is above EO settlement (Form B). This will require the users to refer the case to a caseworker of a higher grade. The referral option then adds a PAS note and returns to Workflow.
