Combinations

When an application contains multiple transactions, a caseworker can currently only work on one of these in Application Processing. This new feature will allow caseworkers to work on 'combinations of transactions'. The piece of work is starting with just Charge and RX1 transactions for Phase 1, Phase 2 will include Subsequent Discharge, and Phase 3 will add Transfers.

A caseworker:

  • will be able to process a combination of transactions in Application Processing
  • can skip a transaction without dropping out of Application Processing
  • have sight of what transactions are included in the application
  • have sight of what has happened with the application before submitting

The system will:

  • save drafting on completed transactions
  • identitfy transactions and when these can or cannot be processed in Application Processing
  • group transactions level tasks, as well as initial and final tasks as 'pseudo transactions'

Phase 1 prototype

After severla rounds of user testing and iterations, combinations phase 1 is now live. This prototype documents the design of the combinations feature. User research, iterations and design history for this piece of work can be found in the old AP prototype kit under 'multi-transactions'

Design considerations:

Link to prototype flow

  • A - Both UI tasks
    Scenario A: The register contains a charge with a restriction and another type of restriction. The other type of restriction does not require consent. The charge evidence is missing. The application should defer with the charge evidence, however our assumption is that it won't due to both tasks being present and caseworkers answering with workarounds.
  • B - Two restrictions
    Scenario B: One restriction requires consent, one does not. Consent has not been provided for the restriction that requires it and the case should therefore defer.
  • C - Multiple restrictions
    Scenario C: Multiple restrictions, some needing consent and others not. Consent has been provided for all that require it and therefore the case should markoff.
  • D - Referral
    Scenario D: One restriction that is above EO settlement (Form B). This will require the users to refer the case to a caseworker of a higher grade. The referral option then adds a PAS note and returns to Workflow.
